# Future Ideas

This document captures ideas that may become valuable later, but are intentionally deferred until the operational core is solid.

Deferring these ideas is a discipline choice, not a sign that they lack value.

## Integrations

### Delivery app integrations

Possible future integrations with delivery or marketplace platforms could reduce duplicate order entry.

Deferred because the core value of SKOSS should first work even without external platform connections.

### WhatsApp integration

Potential uses:

- order intake from messages
- confirmation flows
- customer updates
- internal operational notifications

Deferred because messaging workflows can expand scope quickly and should rest on a stable order and recurrence model first.

## Advanced item identification

### Barcode workflows

Useful later for packing, verification, or stock-related processes.

Deferred because barcode-driven flows should remain optional and should not shape the initial operator experience.

### SKU/internal code workflows

Useful for advanced catalog management, integrations, and reporting.

Deferred because many small kitchens can operate effectively without making internal codes central to daily work.

## Automation and rules

### Automation rules

Possible future capabilities:

- auto-generate prep tasks from order thresholds
- notify a supervisor when a recurring order changes
- trigger reminders for fermentation or proofing windows

Deferred because rule systems can become complex quickly. The core manual workflow should work well first.

### Visual node-based process builder

A future visual builder could let advanced users define process logic more flexibly.

Deferred because early product success depends on a clear default model, not a flexible but heavy meta-tool.

## Analytics and reporting

### Expanded analytics

Possible future areas:

- product trend views
- demand variability
- route performance
- waste and shortfall analysis
- operator throughput summaries

Deferred because the project should first make execution clearer before investing heavily in reporting breadth.

## Inventory depth

### Deeper inventory features

Possible future areas:

- ingredient stock tracking
- lot/batch traceability
- purchasing support
- supplier workflows
- variance analysis

Deferred because deep inventory systems can dominate product design and pull the project toward ERP territory too early.

## Presets and presentation layers

### Skin/preset system for different kitchen types

A long-term direction is a shared operational core with different presets or skins for:

- bakery
- dark kitchen
- food stall
- prep kitchen
- small restaurant

This could affect vocabulary, default workflows, and visible modules.

Deferred because the bakery-first domain needs to prove the foundation before generalizing.

## Broader platform ideas

Other future possibilities may include:

- offline-first support for unstable connectivity environments
- richer dispatch and route coordination
- printable production sheets and labels
- multi-location coordination
- richer permissions and audit history

These are worth keeping in mind, but they should not dilute the first operational milestone.
