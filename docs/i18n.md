# Internationalization (i18n)

## Purpose

SKOSS should treat multi-language support as a core product capability for SKOSSina, not as a late-stage UI enhancement.

The goal is to make the system practical for mixed-language food teams while keeping the core architecture simple and maintainable.

## Launch language support

The launch language set should include at least:

- English (`en`)
- Spanish (`es`)
- Portuguese (`pt`)

Additional languages may be added later as dictionaries and review capacity grow.

## Core architectural stance

A few rules should stay fixed:

- **SKOSS core is language-agnostic.** It stores stable operational data, identifiers, statuses, and configuration references rather than rendered UI strings.
- **SKOSSina handles presentation.** The client resolves translation keys into human-readable labels, messages, prompts, and workflow text.
- **Stored data is not translated in place.** Changing the UI language must not mutate operational records.
- **Preset terminology is layered on top of translations.** A preset may prefer one translated term over another, but it should not require a separate business-logic branch.

## Language layers

SKOSS should distinguish clearly between three different language layers.

### 1. System language

This is the language of the interface itself. It includes:

- navigation labels
- button text
- status labels
- form help text
- validation and empty-state messages
- workflow prompts and action names

This layer must be fully key-based.

### 2. Preset or business terminology

This is the language style chosen for a business type or starting template. It includes:

- whether the UI says `work item`, `prep item`, or another preferred term
- default names for stations, destinations, or workflow views
- wording emphasis that fits bakery, cafe, restaurant, or dark-kitchen usage

This layer should be configurable through presets and dictionaries, not hardcoded into the core model.

### 3. User-entered content

This is real content entered by people using the system, such as:

- customer names
- freeform notes
- order comments
- item nicknames
- destination notes

This content should be stored exactly as entered. It is not automatically translated.

## Translation key approach

UI text should always be referenced through stable translation keys rather than inline strings.

Preferred pattern:

- component requests a translation key such as `orders.list.title`
- active dictionary returns the localized string
- optional preset terminology can override a subset of keys or values for a business type

Avoid:

- hardcoded strings inside components
- storing translated UI labels in business records
- branching business logic by language

## Suggested dictionary structure

The translation system should stay simple and readable. A practical starting structure is:

```text
i18n/
  en/
    common.json
    orders.json
    production.json
    handoff.json
    presets.json
  es/
    common.json
    orders.json
    production.json
    handoff.json
    presets.json
  pt/
    common.json
    orders.json
    production.json
    handoff.json
    presets.json
```

This keeps dictionaries grouped by language and feature area without introducing a heavy localization platform too early.

## How UI components should consume translations

Components should follow a few simple rules:

1. request strings through a translation function or hook using keys
2. keep display text outside business logic
3. allow preset terminology to override presentation where appropriate
4. fall back predictably when a key is missing

A simplified example:

```ts
const title = t('orders.list.title')
const workLabel = t('production.work_item.label')
```

If a bakery preset wants `production.work_item.label` to appear as `batch` in one language and a cafe preset wants `prep item`, the preset layer should influence the returned term without changing the underlying data model.

## Translation dictionaries and presets

Presets should work with the i18n system, not around it.

A preset may define:

- terminology overrides
- default feature labels
- example content translated per supported language
- recommended workflow wording

But presets should not:

- replace core entity semantics
- fork application logic by business type
- require separate data models for each language

## Stored data model implications

The data model should remain stable regardless of UI language. That means:

- identifiers and status codes should stay language-neutral
- records should not store display strings when a stable code or key is enough
- translations should happen at read and render time in the client
- exports and APIs should prefer stable values over localized labels

This protects portability, reporting, and future language additions.

## Future language expansion strategy

Language expansion should remain straightforward:

1. add a new language code
2. create dictionaries for the same key set
3. review terminology coverage for existing presets
4. verify fallback behavior for any missing keys
5. ship without changing the core schema

This process should make it possible to add languages incrementally without blocking the rest of product development.

## Practical guardrails

To keep the i18n system useful without over-engineering it:

- start with key-based dictionaries and a small number of language files
- keep interpolation simple and explicit
- avoid building a complex translation management platform too early
- prefer stable keys that match feature structure
- review new UI work for hardcoded strings before merge

The goal is a simple, scalable foundation that supports English, Spanish, and Portuguese well and leaves a clear path for more languages later.
