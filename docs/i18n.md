# Internationalization (i18n)

## Purpose

SKOSS should treat multi-language support as a core product capability for SKOSSina, not as a late-stage UI enhancement.

The goal is to make the system practical for mixed-language food teams while keeping the core architecture simple and maintainable.

## Launch language support

The launch language set should include at least:

- English (`en`)
- Spanish (`es`)
- Portuguese (`pt`)

Additional languages may be added later as dictionaries and review capacity grow.

## Core architectural stance

A few rules should stay fixed:

- **SKOSS core is language-agnostic.** It stores stable operational data, identifiers, statuses, and configuration references rather than rendered UI strings.
- **SKOSSina handles presentation.** The client resolves translation keys into human-readable labels, messages, prompts, and workflow text.
- **Stored data is not translated in place.** Changing the UI language must not mutate operational records.
- **Preset terminology is layered on top of translations.** A preset may prefer one translated term over another, but it should not require a separate business-logic branch.

## Language layers

SKOSS should distinguish clearly between three different language layers.

### 1. System language

This is the language of the interface itself. It includes:

- navigation labels
- button text
- status labels
- form help text
- validation and empty-state messages
- workflow prompts and action names

This layer must be fully key-based.

### 2. Preset or business terminology

This is the language style chosen for a business type or starting template. It includes:

- whether the UI says `work item`, `prep item`, or another preferred term
- default names for stations, destinations, or workflow views
- wording emphasis that fits bakery, cafe, restaurant, or dark-kitchen usage

This layer should be configurable through presets and dictionaries, not hardcoded into the core model.

### 3. User-entered content

This is real content entered by people using the system, such as:

- customer names
- freeform notes
- order comments
- item nicknames
- destination notes

This content should be stored exactly as entered. It is not automatically translated.

## Translation key approach

UI text should always be referenced through stable translation keys rather than inline strings.

Preferred pattern:

- component requests a translation key such as `orders.list.title`
- active dictionary returns the localized string
- optional preset terminology can override a subset of keys or values for a business type

Avoid:

- hardcoded strings inside components
- storing translated UI labels in business records
- branching business logic by language

## Current implementation

The current codebase uses a lightweight internal i18n layer under `lib/i18n/`.

### Files

```text
lib/i18n/
  config.ts
  index.ts
  server.ts
  dictionaries/
    en.ts
    es.ts
    pt.ts
components/
  i18n-provider.tsx
  language-switcher.tsx
```

### Runtime flow

1. the active locale is read from the `skoss-locale` cookie on the server
2. the root layout builds the translator and wraps the app in `I18nProvider`
3. server components call `getServerTranslator()`
4. client components call `useI18n()`
5. if a key is missing in the active locale, the system falls back to English
6. if a key is missing entirely, the raw key string is returned so the gap is obvious during development

### Language preference persistence

The current implementation keeps language preference lightweight:

- the user/preferences menu language selector updates `skoss-locale`
- the same value is also written to `localStorage`
- a small `beforeInteractive` layout script restores the language cookie from local storage if needed

This keeps the current app simple without introducing a heavier localization framework.

## Preset terminology hook

Preset terminology is handled inside `lib/i18n/index.ts` by merging locale dictionaries with preset-specific overrides.

Current supported presets:

- `generic`
- `bakery`
- `cafe`
- `small_restaurant`
- `dark_kitchen`
- `food_stall`
- `other`

The current default is `generic`, with the onboarding assistant guiding the first selection so teams can start from a closer vocabulary and workspace emphasis without being locked into it.

Examples of preset-aware terminology keys:

- `terms.workItem.one`
- `terms.workItem.many`
- `terms.destination.one`
- `terms.destination.many`
- `terms.handoff`
- `terms.fulfillment`
- `terms.productionDay`

## Onboarding assistant and preset behavior

The first-run onboarding assistant now collects only a small set of practical preferences:

- business name
- interface language
- business-type preset
- operating mode
- appearance preference

These selections are intentionally lightweight. Today they change:

- the active interface locale
- preset terminology overrides where supported by `terms.*`
- the initial emphasis and ordering of home workspace links so first-run navigation stays closer to the selected preset
- starter suggestions and example setup copy shown in Home and Setup
- the saved theme and business name used by the shell

They do **not** yet create a separate domain model, hard-fork workflows, or enforce setup rules by business type.

UI code should consume these terms through the translator instead of baking preset vocabulary directly into components.

## How to add new strings

When adding UI text:

1. choose a stable feature-oriented key
2. add it to `lib/i18n/dictionaries/en.ts`
3. add matching entries to `es.ts` and `pt.ts`
4. use `t('your.key')` in server or client UI code
5. if the text is terminology-sensitive, prefer a `terms.*` key or a preset override instead of a bakery-only label

A few practical rules:

- prefer grouped keys by feature area such as `orders.*`, `production.*`, `handoff.*`, `setup.*`
- keep interpolation simple, using `{{name}}` placeholders only where needed
- do not translate user-entered business content in persistence or server actions
- do not store rendered labels when a stable code already exists

## Formatting guidance

Locale-sensitive formatting should happen at render time.

Current examples:

- date formatting uses the active locale in `lib/domain/formatters.ts`
- status and shift labels can be resolved through translation keys
- recurring schedule labels use localized weekday abbreviations

## Future evolution

Near-term next steps should likely include:

- moving more remaining UI literals onto translation keys as features evolve
- extending metadata and document titles to use locale-aware labels where helpful
- introducing workspace- or setup-level preset selection once presets become real product configuration
- adding lightweight translation coverage checks so missing keys are easier to catch in CI

The goal remains a simple, understandable foundation that keeps SKOSS core neutral while letting SKOSSina present the right language and terminology for each operator context.
